# SHOT - AI 테니스 경기 분석 앱 설계 문서

## 개요

테니스 경기 중 코트 뒤편에 핸드폰을 거치하여 영상을 촬영하면, AI가 코트 라인을 인식하고 공의 IN/OUT을 판정하며, 스코어를 자동으로 기록하는 앱.

### 참고 서비스
- Hawk-Eye (프로 경기 공식 시스템) - 다중 카메라 삼각측량 방식
- SwingVision (iOS 앱) - 단일 카메라 AI 분석 방식
- **중요**: 위 서비스의 특허/저작권을 침해하지 않는 독자적 구현 필수

### 배경
- 이전 DEEP SHOT 프로젝트에서 ML 기반 **코트 라인 세그멘테이션** 방식으로 시도했으나, 코트 라인 검출에 실패하여 프로젝트 중단
- 이번에는 **키포인트 검출 + 호모그래피 역산** 방식으로 접근하여 이전 실패를 극복
- 코트 라인을 픽셀 단위로 찾는 대신, 코트의 구조적 교차점(키포인트)을 찾아 전체 코트를 추론

## Phase 2: 공 추적 & IN/OUT 판정 로드맵

> Phase 1 코트 인식이 안정화된 후 진행. 코트 좌표계가 확립되어야 공의 착지 위치를 판정할 수 있음.

### 2-1. 핵심 목표

| 목표 | 설명 | 성공 기준 |
|------|------|----------|
| 공 검출 | 영상 프레임에서 테니스공 위치 검출 | 바운스 프레임의 90%+ 검출 |
| 궤적 추적 | 프레임간 공 위치 연결하여 궤적 생성 | 연속 5프레임 이상 추적 유지 |
| 바운스 감지 | 공이 코트에 닿는 순간 감지 | 바운스의 85%+ 감지 |
| IN/OUT 판정 | 바운스 위치가 라인 안/밖 판정 | 정확도 90%+ (동호인 수준) |
| 실시간 알림 | 판정 결과를 시각적으로 표시 | 바운스 후 1초 이내 표시 |

### 2-2. 기술 접근법

#### 방법 A: 전용 공 검출 모델 (권장)

```
[카메라 프레임]
    ├── [코트 인식 모델] → 코트 좌표계 (Phase 1)
    │
    └── [공 검출 모델] → 공 위치 (px)
            │
            ▼
        [궤적 추적기] → 연속 궤적
            │
            ▼
        [바운스 감지] → 착지 프레임 특정
            │
            ▼
        [좌표 변환] → 공 위치를 코트 좌표로 변환 (Phase 1의 H 사용)
            │
            ▼
        [IN/OUT 판정] → ITF 규격 라인과 비교
```

**공 검출 모델: TrackNet (확정)**

YOLO 계열은 작은 공(3-15px) 검출에 약하고 단일 프레임만 사용하여 모션 블러에 취약.
TrackNet은 연속 3프레임의 시간 정보를 활용하여 작은 공과 빠른 움직임에 강함.

**TrackNet 기반 접근**
- 입력: 연속 3프레임 (t-2, t-1, t) 합쳐서 [H, W, 9] 텐서
- 출력: 256×256 히트맵 (공 위치 확률)
- 장점: 작은 공, 모션 블러, 가려짐에 강함
- 모델 크기: ~3MB (TFLite, FP16)

#### 방법 B: 하이브리드 (실시간 + 후처리)

실시간으로는 간단한 모션 감지만 수행하고, 영상 저장 후 정밀 분석:
- **실시간**: 프레임 차분 기반 빠른 공 위치 추정 → 대략적 IN/OUT
- **후처리**: 저장된 영상에서 TrackNet으로 정밀 분석 → 정확한 판정

동호인 사용자에게는 경기 후 리뷰가 더 실용적일 수 있음.

### 2-3. 바운스 감지 알고리즘

```
공 궤적: y좌표 변화 패턴으로 바운스 감지

    y좌표   ↑ (화면 위)
            │   ╲         ╱╲
            │    ╲       ╱  ╲
            │     ╲     ╱    ╲
            │      ╲   ╱      ╲
            │       ╲ ╱        → 바운스 지점
            │        V           (y 방향 반전)
            └──────────────────→ 시간

바운스 조건:
1. y좌표가 감소→증가로 전환 (화면에서 아래→위)
2. 전환점의 y좌표가 코트 영역 내 (호모그래피 변환 후 확인)
3. 공 크기가 일정 범위 내 (너무 크면 선수, 너무 작으면 노이즈)
```

### 2-4. IN/OUT 판정 로직

```kotlin
// 바운스 위치를 코트 좌표(미터)로 변환
val courtPos = homography.projectImageToCourt(bounceX, bounceY, H)

// ITF 규격 라인과 비교
val isIn = when (shotType) {
    SERVE_DEUCE -> courtPos.isInServiceBox(Side.DEUCE)
    SERVE_AD -> courtPos.isInServiceBox(Side.AD)
    RALLY -> courtPos.isInSinglesOrDoublesCourt()
}

// 판정 기준: 공의 외곽이 라인에 닿으면 IN
// 테니스공 직경 6.54-6.86cm → 반지름 ~3.4cm
// 공 중심이 라인 밖이라도 3.4cm 이내면 IN
val BALL_RADIUS = 0.034f // meters
val marginAdjustedPos = adjustForBallRadius(courtPos, BALL_RADIUS)
```

### 2-6. 핸드폰 카메라의 한계와 대응

| 한계 | 영향 | 대응 |

## Phase 1: 코트 인식 엔진 상세 설계

### 1. 아키텍처

```
[CameraX ImageAnalysis] ─── 화면 가로(landscape) 고정
        │
        ▼
[프레임 캡처 (최신 프레임만 처리, 버퍼 >2시 이전 프레임 드롭)]
        │
        ▼

        │

        │
        ▼

        │
        ▼
[적응형 키포인트 필터링 (신뢰도 기반, 최소 6개 필수)]
        │
        ▼
[호모그래피 행렬 계산 + 검증]
        │
        ▼
[시간적 평활화 (지수 이동 평균, alpha=0.7)]
        │
        ▼
[전체 코트 좌표 투영 (16개 키포인트)]
        │
        ▼
[화면에 코트 라인 오버레이 렌더링 + 신뢰도 시각화]
```

### 3. 코트 키포인트 정의

표준 테니스 코트 (ITF Rules of Tennis 2024, Section 1):
- 코트 길이: 23.77m (78ft)
- 단식 폭: 8.23m (27ft)
- 복식 폭: 10.97m (36ft)
- 서비스라인까지: 베이스라인에서 6.40m (21ft)
- 네트까지: 베이스라인에서 11.885m
- 네트 높이: 중앙 0.914m (3ft)
- **라인 폭: 2.5~5cm** (IN/OUT 판정 시 중요)

```

라인  라인                          라인  라인
 |     |            |               |     |
 1─────2────────────3───────────────4─────5   ← 베이스라인 (원코트)
 |     |            |               |     |
 |     6────────────7───────────────8     |   ← 서비스라인 (원코트)
 |     |            |               |     |
 |     |          [NET]             |     |   ← 네트
 |     |            |               |     |

                  📱 카메라 위치 (코트 뒤편)
```

**총 16개 키포인트 중 ML 모델이 검출하는 포인트: 최대 8개 (니어코트)**

#### 가시 키포인트 (ML 모델 검출 대상)

카메라가 코트 뒤편(니어코트 베이스라인 뒤)에 위치하므로, 원코트(1~8번)는 보이지 않음.

| 우선순위 | # | 위치 | 비고 |
|---------|---|------|------|
| **필수** | 9 | 니어 서비스라인 × 단식 좌측 사이드라인 | 항상 카메라에 보임 |

### 4. 적응형 키포인트 검출 전략

```
ML 모델 출력: 8개 키포인트 좌표 + 각각의 신뢰도 점수 (0.0~1.0)

CASE 1: 8개 모두 신뢰도 >= 임계값(0.7)
    → 8개 포인트 모두 활용하여 호모그래피 계산 (최대 정확도)

CASE 2: 필수 6개(9,10,11,13,14,15) 신뢰도 >= 임계값, 12/16 미달
    → 필수 6개만으로 호모그래피 계산
    → 복식 라인(12,16)은 호모그래피로 역산

CASE 3: 유효 포인트 < 6개
    → "코트를 인식할 수 없습니다" 경고 표시
    → 카메라 위치/각도 조정 가이드 제공
    → 호모그래피 계산하지 않음 (이전 유효 프레임의 호모그래피 유지)

급격한 변화 감지:
    → 연속 프레임 간 키포인트 이동 > 20px → 평활화 필터 리셋 (장면 변화)
    → 선수가 코트 라인을 가리는 경우 → 호모그래피 업데이트 일시 중단
```

**최소 6개 포인트 요구 이유**: 호모그래피는 이론적으로 4개 대응점이면 계산 가능하나, 4~5개에서는 원코트 키포인트(10m+ 거리) 투영 시 오차가 심하게 증폭됨. 6개 이상의 과결정계(overdetermined system)에서만 안정적인 결과를 얻을 수 있음.

### 5. 렌즈 왜곡 보정

스마트폰 카메라(특히 광각 24mm 이하)의 배럴 왜곡(10~15%)이 키포인트 검출 정확도에 영향:

- **방법 1 (추천)**: 학습 데이터에 다양한 렌즈 왜곡을 포함하여 모델이 왜곡 자체를 학습
  - 왜곡된 이미지에서 직접 키포인트를 예측하도록 학습
  - 추가 전처리 비용 없음
- **방법 2 (대안)**: OpenCV `undistort()`로 프레임을 보정한 후 ML 모델에 입력
  - 카메라 내부 파라미터(focal length, distortion coefficients) 필요
  - CameraX에서 카메라 특성 자동 획득 가능
- **권장 지원 범위**: 표준 화각(24~28mm) 최적, 울트라와이드(16mm)는 정확도 저하 가능 → 앱에서 경고 표시

### 6. 카메라 적응형 처리

사용자마다 삼각대 높이/각도가 다르므로, AI가 자동 적응:

- **호모그래피가 카메라 파라미터를 자동으로 보상**: 어떤 각도/높이에서든 키포인트만 검출되면 정확한 코트 좌표 매핑 가능
- **초기 캘리브레이션 불필요**: 키포인트 검출 + ITF 표준 치수만으로 호모그래피 계산 (렌즈 왜곡은 학습 데이터로 보상)
- **카메라 위치 가이드**: 코트 인식 품질이 낮을 때 구체적 가이드 표시
  - "카메라를 더 높이 올려주세요"
  - "카메라를 코트 중앙에 맞춰주세요"
  - "울트라와이드 모드를 끄세요"
- **시간적 평활화**: 지수 이동 평균(EMA, alpha=0.7)으로 호모그래피 행렬 평활화
  - 출력 지연: 약 1~2프레임 (33~66ms), 허용 범위 내
  - 키포인트 급변(>20px) 감지 시 필터 리셋

### 7. 호모그래피 계산 상세

**수학적 정의**: 호모그래피 H는 코트 평면(실제 좌표)과 이미지 평면(픽셀 좌표)을 매핑하는 3×3 행렬.

1. **입력**: ML 모델로 검출된 가시 키포인트의 이미지 좌표 `p_image` (픽셀)
2. **대응점**: ITF 규격 표준 코트의 실제 좌표 `p_court` (미터 단위)
3. **호모그래피 방향**:
   - `p_court = H × p_image` (이미지 → 실제 코트 좌표)
   - `p_image = H⁻¹ × p_court` (실제 코트 좌표 → 이미지, 오버레이용)
   - OpenCV `findHomography(srcPoints=p_image, dstPoints=p_court)` 로 H 계산
4. **RANSAC**: 이상치 제거를 위해 `findHomography(..., method=RANSAC)` 사용
5. **역투영 (오버레이용)**: 보이지 않는 원코트 키포인트의 ITF 좌표를 `H⁻¹`로 이미지 좌표에 투영
6. **검증 절차**:
   - 가시 키포인트 재투영 오차 확인: `|p_detected - H⁻¹ × p_court| < 5px`
   - 역산된 원코트 포인트(1~8)가 이미지 프레임 합리적 범위 내 (±1000px)
   - 역산 결과가 기하학적으로 유효한지 확인 (예: 1번이 5번보다 왼쪽에 있는지)
   - 검증 실패 시 → 해당 프레임 호모그래피 폐기, 이전 유효 프레임 유지

### 8. 오차 전파 분석 (Phase 2 의존성)

Phase 2(IN/OUT 판정)에서의 정확도는 Phase 1의 키포인트 검출 정확도에 직접 의존.
ITF 라인 폭은 2.5~5cm이므로, 코트 좌표 오차가 이 범위를 넘으면 오판 발생.

| 키포인트 검출 오차 (px) | 니어코트 실제 오차 (~5m) | 원코트 실제 오차 (~20m) |
|----------------------|----------------------|----------------------|
| ±1px | ±0.5cm | ±2cm |
| ±2px | ±1cm | ±4cm |
| ±3px | ±1.5cm | ±6cm |
| ±5px | ±2.5cm | ±10cm |

**결론**: 원코트(원거리) IN/OUT 판정의 정확도를 위해:
- 니어코트 키포인트(13,14,15) 오차 목표: **< 2px** (원코트 역산에 가장 큰 영향)
- 서비스라인 키포인트(9,10,11) 오차 목표: **< 3px**
- 복식 사이드라인 키포인트(12,16) 오차 목표: **< 4px**

### 9. ML 모델 학습 전략

- **아키텍처**: MobileNet V2/V3 백본 + 키포인트 회귀 헤드 (경량, 모바일 최적화)
  - 입력: 256×256 RGB 이미지
  - 출력: 8개 키포인트 × (x, y, confidence) = 24개 값
  - **TFLite 호환 연산자만 사용** (변환 실패 방지)
  - 목표 모델 크기: FP32 < 10MB, INT8 양자화 < 5MB (정확도 손실 < 2%)

- **학습 데이터 확보**:
  - 공개 데이터셋 활용 (Kaggle Tennis Court Detection datasets)
  - 직접 촬영: 다양한 코트 표면(하드/클레이/잔디/카펫), 실내/실외, 다양한 조명
  - **카메라 위치 다양성**: 높이 0.5~3.5m, 수평 각도 0~45°, 앙각 20~60°
  - **렌즈 다양성**: 표준(24~28mm), 광각(16~24mm) 화각 포함
  - 데이터 증강 (확률 기반):
    - 밝기 ±20% (p=0.5), 대비 ±15% (p=0.4), 색조 ±10° (p=0.3)
    - 회전 ±5° (p=0.2), 크롭 5~10% (p=0.3)
    - 합성 그림자 (p=0.2), 렌즈 왜곡 시뮬레이션 (p=0.3)

- **학습 파이프라인**: Python + PyTorch → TFLite 직접 변환 (ONNX 경유 시 호환성 문제 방지)
  - 변환 파이프라인을 모델 학습 전에 먼저 테스트하여 호환성 확인

- **평가 지표 (키포인트별 차등)**:
  - 베이스라인 키포인트(13,14,15): 평균 오차 < 2px
  - 서비스라인 키포인트(9,10,11): 평균 오차 < 3px
  - 복식 사이드라인(12,16): 평균 오차 < 4px
  - 신뢰도 정밀도/재현율
  - 호모그래피 재투영 오차 (미터 단위)

- **목표 성능**:
  - 주 타겟(Snapdragon 8 Gen 2): 추론 < 30ms
  - 최소 타겟(Snapdragon 6 Gen 1): 추론 < 80ms
  - 20개 다양한 테스트 영상에서 키포인트 정확도 달성

### 10. Phase 1 MVP 화면 구성

**화면 방향: 가로(landscape) 고정** (회전 시 호모그래피 좌표 깨짐 방지)

1. **메인 카메라 화면**
   - 카메라 실시간 미리보기 (전체 화면)
   - 검출된 코트 라인 오버레이 (초록색 선, 니어코트 + 역산된 원코트)
   - **키포인트 신뢰도 시각화**: 각 키포인트를 원으로 표시
     - 초록(>0.8): 정확, 노랑(0.6~0.8): 불안정, 빨강(<0.6): 미검출
   - 코트 인식 상태 인디케이터 (인식됨 ✓ / 미인식 ✗)
   - 녹화 시작/중지 버튼
   - 카메라 위치 가이드 (인식 품질 낮을 때만 표시)

2. **설정 화면**
   - 카메라 해상도 선택
   - 모델 정밀도 모드 (빠름/정확)
   - 단식/복식 모드 선택
   - 언어 설정 (한국어/영어)

3. **디버그 모드** (개발용, 설정에서 활성화)
   - 키포인트 좌표 + 신뢰도 점수 표시
   - 검출 위치(초록 원) vs 재투영 위치(빨강 원) 비교 → 호모그래피 오차 시각화
   - 추론 시간 (ms) 표시
   - 호모그래피 재투영 오차 (px) 표시
   - FPS 카운터
