# 프로젝트 오답노트 - 테니스 코트 키포인트 검출

> 이 문서는 프로젝트 진행 중 겪은 실패, 원인 분석, 외부 피드백, 그리고 향후 방향을 정리한 "오답노트"입니다.
> 같은 실수를 반복하지 않기 위해, 그리고 미래의 의사결정에 근거를 제공하기 위해 작성합니다.
>
> 최종 갱신: 2026-03-18

---

## 1. 문제 정의

- **목표**: 핸드폰 카메라로 촬영한 테니스 코트 이미지에서 8개 키포인트(코트 꼭짓점 등) 검출
- **배포 환경**: Android 모바일 (TFLite 변환 필요)
- **성능 목표**: 평균 오차 2~4px 이내 (256×256 입력 기준)
- **모델 제약**: MobileNetV3-Small 급 경량 백본 사용 (모바일 추론 속도 확보)

### 핵심 챌린지

| 항목 | 설명 |
|------|------|
| 도메인 갭 | 학습 데이터(방송)와 실제 사용 환경(핸드폰)의 큰 차이 |
| 데이터 부족 | 타겟 도메인(핸드폰) 데이터가 339장뿐 |
| 모바일 제약 | 모델 크기·속도 제한으로 고성능 백본 사용 불가 |
| 키포인트 특성 | 가려지거나 프레임 밖에 있는 키포인트 처리 필요 |

## 2. 시도한 것들과 결과

### 시도 1: 방송 데이터(8,841장)로만 학습

- **모델**: MobileNetV3-Small + FC direct regression
- **학습 데이터**: 방송 프레임 8,841장 (고정 카메라, 높은 앵글, 고화질)
- **결과**:
  - 방송 테스트셋: 평균 오차 **4~7px** → 성공으로 보였음
  - 핸드폰 테스트셋(67장): 평균 오차 **37.06px** → **완전 실패**

#### 교훈

> **같은 도메인에서의 테스트는 의미가 없다.**
> 소스 도메인에서 아무리 좋은 성능이 나와도, 타겟 도메인에서 검증하지 않으면 실제 성능을 알 수 없다.
> 프로젝트 초기부터 타겟 도메인 테스트셋을 확보하고, 모든 실험의 평가 기준으로 삼아야 한다.

#### 놓친 것

- 방송 카메라와 핸드폰 카메라의 차이를 과소평가
- "일단 방송 데이터로 잘 되면 핸드폰도 되겠지"라는 근거 없는 낙관
- 타겟 도메인 테스트셋을 프로젝트 시작 시점에 준비하지 않음

---

### 시도 2: 핸드폰 YouTube 데이터(339장) 수집 & 라벨링

- **데이터 수집**: YouTube에서 동호인 테니스 영상 검색 → 프레임 추출 → 수동 라벨링
- **수집 파이프라인**: `yt-dlp` → OpenCV 프레임 추출 → 커스텀 라벨링 도구
- **학습 데이터**: 핸드폰 데이터만 272장 (train), 67장 (test)
- **결과**: 평균 오차 **24.03px** (목표 2~4px에 한참 미달)

#### 교훈

> **339장은 턱없이 부족하다.**
> 딥러닝 키포인트 검출에서 클래스당 최소 1,000장은 되어야 "어라 좀 되네?" 수준이 시작된다.
> 데이터 수집과 라벨링에 시간을 아까워하면, 모델 튜닝에 훨씬 더 많은 시간을 낭비하게 된다.

#### 놓친 것

- 라벨링 시간 대비 효과를 고려하지 않음 (339장 라벨링에 시간을 들이고도 부족)
- 모델 예측을 보조로 활용한 반자동 라벨링을 초기에 도입하지 않음
- 데이터 다양성 검토 미흡 (코트 종류, 앵글, 조명 조건의 분포 분석 없음)

---

### 시도 3: 방송 + 핸드폰 데이터 단순 합산

- **학습 데이터**: 방송 1,000장 + 핸드폰 272장 = 총 1,272장
- **결과**: 평균 오차 **24.95px** → 핸드폰만(24.03px)보다 **오히려 나빠짐**

#### 교훈

> **단순 합산은 도메인 갭을 해결하지 못한다. 오히려 악화시킬 수 있다.**
> 방송 데이터가 양적으로 압도하면서, 모델이 방송 도메인의 분포에 맞춰 학습됨.
> 핸드폰 데이터가 "소수파 노이즈"처럼 취급되어 오히려 학습을 방해.

#### 놓친 것

- 두 도메인의 데이터를 동등하게 섞으면 안 된다는 점
- 타겟 도메인 오버샘플링의 필요성
- pretrain → mixed → fine-tune 3단계 전략의 부재
- 도메인별 배치 구성 전략 미적용

## 3. 근본 원인 분석

### 3-1. 아키텍처 문제: FC Direct Regression의 한계

```
[문제가 된 구조]
이미지(256×256) → MobileNetV3-Small → Global Average Pooling → FC → 좌표값(x1,y1,...,x8,y8)
```

- **공간 정보 완전 손실**: Global Average Pooling이 feature map의 위치 정보를 모두 버림
- **직접 좌표 회귀의 불안정성**: 연속적인 좌표값을 직접 예측하는 것은 비선형 매핑이 극도로 어려움
- **히트맵 기반 방식이 우월한 이유**:
  - feature map에서 각 키포인트의 위치를 확률 분포(가우시안 히트맵)로 예측
  - 공간 정보가 보존되어 정밀도가 구조적으로 높음
  - argmax 또는 soft-argmax로 서브픽셀 정확도 달성 가능

> **결론**: FC direct regression은 이 태스크에 구조적으로 부적합.
> 히트맵 head로의 전환이 가장 큰 성능 점프를 가져올 것으로 예상.

### 3-2. 데이터 부족

| 측면 | 현재 상태 | 필요 수준 |
|------|----------|----------|
| 타겟 도메인 데이터 수 | 339장 | 최소 1,000장, 이상적으로 3,000장+ |
| 코트 종류 다양성 | 불명확 | 하드/클레이/잔디/실내 등 |
| 촬영 앵글 다양성 | 제한적 | 낮은/중간/높은 앵글, 좌우 다양 |
| 조명 조건 | 제한적 | 주간/야간/실내/역광/그림자 |
| 가려짐 케이스 | 불명확 | 선수에 의한 키포인트 가려짐 포함 |

> **"AI 프로젝트는 데이터가 99%"**
> 데이터가 부족한 상태에서 아키텍처나 하이퍼파라미터를 아무리 바꿔도 한계가 있다.
> 가장 먼저 해야 할 일은 데이터 확보.

### 3-3. 도메인 갭 상세 분석

| 특성 | 방송 카메라 | 핸드폰 카메라 |
|------|-----------|-------------|
| 앵글 | 높은 고정 위치 (관중석 상단) | 낮은 위치 (코트 옆 벤치 등) |
| 렌즈 | 줌 렌즈 (좁은 화각) | 광각 렌즈 (넓은 화각, 배럴 왜곡) |
| 해상도 | 고화질, 일관된 품질 | 다양한 화질, 압축 아티팩트 |
| 배경 | 관중석, 광고판 (일관적) | 나무, 건물, 주차장 등 (다양) |
| 조명 | 조명 시설 (균일) | 자연광 (그림자, 역광 등 다양) |
| 코트 상태 | 깨끗한 프로 코트 | 낡은 라인, 먼지, 물기 등 |
| 안정성 | 삼각대 고정 (흔들림 없음) | 수동 촬영 (흔들림, 기울어짐) |

> **핵심**: 이 모든 차이가 동시에 존재하기 때문에, 단순 augmentation으로는 갭을 메울 수 없다.
> 타겟 도메인의 실제 데이터가 반드시 필요하다.

### 3-4. 학습 전략 미흡

- **단순 A/B/C 비교만 수행**: "방송만", "핸드폰만", "합산" 세 가지만 비교
- **적용하지 못한 전략들**:
  1. pretrain → mixed → fine-tune 3단계 학습
  2. 타겟 도메인 오버샘플링 (배치 내 타겟 비율 조절)
  3. 도메인 적응(domain adaptation) 기법
  4. 커리큘럼 러닝 (쉬운 샘플 → 어려운 샘플 순서)

## 4. 외부 피드백 (다른 개발자 + ChatGPT)

### 4-1. 핵심 조언 요약

| # | 조언 | 상세 |
|---|------|------|
| 1 | **3단계 학습** | pretrain(방송 8.8K) → mixed training(타겟 오버샘플링) → target-only fine-tune |
| 2 | **아키텍처 변경** | FC regression → heatmap head (가장 큰 성능 점프 예상) |
| 3 | **데이터 1,000장+ 확보** | 모델 예측 보조 라벨링으로 효율화 |
| 4 | **강한 augmentation** | perspective, 광각왜곡, blur, exposure, JPEG artifact |
| 5 | **confidence 채널 재검토** | 불명확한 supervision이 학습을 흔들 수 있음 |
| 6 | **입력 해상도 상향** | 256 → 384 또는 512 검토 |

### 4-2. 경험 기반 조언 (실무자 의견)

- **"클래스당 1,000장이 '어라 좀 되네?' 수준의 시작점"**
  - 그 이하에서는 모델이 일반화하기 어려움
  - 특히 키포인트 검출은 위치 정밀도가 중요하므로 더 많은 데이터 필요

- **"edge case는 EDA(탐색적 데이터 분석)로 분석 필요"**
  - 어떤 이미지에서 오차가 큰지 시각화
  - 앵글별, 코트별, 조명별 오차 분포 확인
  - 오차가 큰 케이스의 공통점 파악 → 해당 조건의 데이터 보강

- **"in the wild 환경과 유사하게 데이터 구성 필요"**
  - 실제 사용 환경에서 만날 수 있는 모든 조건을 데이터에 포함
  - 깨끗한 데이터만으로는 실전 성능 보장 불가

- **"백본/loss/하이퍼파라미터 튜닝은 기본 조건이 갖춰진 후의 보정"**
  - 데이터와 아키텍처가 근본적으로 부족한 상태에서 튜닝은 무의미
  - 기반이 갖춰진 후에야 미세 조정의 효과가 나타남

- **"pretrained 모델 활용 여부에 따라 결과가 크게 달라짐"**
  - ImageNet pretrained 백본 사용은 거의 필수
  - 소규모 데이터에서 scratch 학습은 과적합 직행

## 5. 앞으로의 방향 (우선순위)

| 순위 | 작업 | 기대 효과 | 난이도 | 비고 |
|-----|------|----------|--------|------|
| 1 | 핸드폰 데이터 1,000장+ 확보 (모델 예측 보조 라벨링) | 근본적 해결 | 중 (시간 소요) | 라벨링 도구 이미 구축 완료 |
| 2 | 히트맵 기반 head로 교체 | 구조적 정밀도 향상 | 중 | SimpleBaseline 또는 Lightweight HRNet 참고 |
| 3 | 3단계 학습 전략 적용 | 도메인 적응 최적화 | 하 | 코드 수정 소규모 |
| 4 | 강한 augmentation 적용 | source→target 갭 축소 | 하 | Albumentations 활용 |
| 5 | conf 채널 제거 + 해상도 384 | 소폭 개선 | 하 | 빠르게 실험 가능 |

### 작업별 상세 계획

**1. 데이터 확보 (최우선)**
- 기존 모델의 예측을 초기값으로 제공하는 반자동 라벨링 도구 활용
- YouTube 동호인 영상 추가 수집 (다양한 코트, 앵글, 조명)
- 목표: 최소 1,000장, 이상적으로 2,000장+
- 라벨링 품질 관리: 키포인트 간 기하학적 일관성 검증

**2. 히트맵 head 교체**
- 현재 FC head를 deconvolution 기반 히트맵 head로 교체
- 출력: 8채널 히트맵 (각 키포인트 당 가우시안 분포)
- Loss: MSE on heatmap (또는 AWING loss)
- soft-argmax로 서브픽셀 좌표 추출

**3. 3단계 학습 전략**
- Stage 1: 방송 데이터(8.8K)로 pretrain → 백본 + head 초기화
- Stage 2: 방송 + 핸드폰 mixed training, 핸드폰 데이터 3~5배 오버샘플링
- Stage 3: 핸드폰 데이터만으로 fine-tune (작은 learning rate)

**4. 강한 augmentation**
- Perspective transform (시점 변화 시뮬레이션)
- Barrel distortion (광각 렌즈 왜곡 시뮬레이션)
- Motion blur, Gaussian blur
- Random brightness, contrast, exposure
- JPEG compression artifact
- Random crop & resize

| 실험 | 학습 데이터 | 테스트 데이터 | 평균 오차 (px) | 비고 |
|-----|-----------|-------------|---------------|------|
| 초기 방송 학습 | 방송 8,841장 | 방송 이미지 | 4~7 | 같은 도메인이라 높은 정확도 |

### 실험 결과에서 읽을 수 있는 것

1. **A vs B**: 타겟 도메인 데이터가 소량이라도 있으면, 대량의 소스 도메인 데이터보다 낫다 (37px → 24px)
2. **B vs C**: 소스 도메인 데이터를 무작정 섞으면 오히려 성능이 떨어질 수 있다 (24.03 → 24.95)
3. **모든 실험**: 목표(2~4px)에는 한참 미달 → 근본적 변화(데이터 + 아키텍처) 필요

## 7. 핵심 교훈

### 교훈 1: 테스트는 반드시 타겟 도메인으로

> 소스 도메인에서의 성능이 아무리 좋아도, 실제 사용 환경에서의 성능과는 무관하다.
> 프로젝트 시작 시점에 타겟 도메인 테스트셋을 확보하고, 이것을 유일한 평가 기준으로 삼아야 한다.

**적용 방법**: 새 프로젝트 시작 시 가장 먼저 타겟 환경 데이터 50~100장 수집 → 테스트셋으로 고정

### 교훈 2: 데이터가 부족하면 아키텍처/전략을 아무리 바꿔도 한계

> "AI 프로젝트는 데이터가 99%"라는 말은 과장이 아니다.
> 모델 구조를 바꾸고, 학습률을 조정하고, augmentation을 추가하는 것은
> 충분한 데이터가 있을 때 의미가 있다.

**적용 방법**: 최소 데이터 요구량(키포인트 검출의 경우 1,000장+)을 사전에 파악하고, 데이터 확보를 최우선 과제로 설정

### 교훈 3: 단순 합산보다 단계적 학습

> 서로 다른 도메인의 데이터를 단순히 섞으면, 양이 많은 쪽에 편향된다.
> pretrain → mixed(오버샘플링) → fine-tune 순서로 단계적으로 학습해야 한다.

**적용 방법**: 도메인이 다른 데이터를 합칠 때는 반드시 오버샘플링 비율과 학습 단계를 설계

### 교훈 4: 직접 회귀보다 히트맵

> 좌표값을 직접 회귀하는 것은 공간 정보를 버리는 것이다.
> 히트맵 기반 방식은 공간 정보를 보존하여 구조적으로 더 정확하다.

**적용 방법**: 키포인트/랜드마크 검출 태스크에서는 히트맵 기반 방식을 기본으로 선택

### 교훈 5: 도메인 갭은 augmentation만으로 해결 불가

> augmentation은 도메인 갭을 "줄여줄" 수는 있지만 "해결"하지는 못한다.
> 타겟 도메인의 실제 데이터가 반드시 필요하다.

**적용 방법**: augmentation은 보조 수단으로 활용하되, 타겟 도메인 데이터 확보에 주력

## 부록: 자주 하는 실수 체크리스트

새 실험을 시작하기 전에 아래 항목을 확인:

- [ ] 타겟 도메인 테스트셋이 준비되어 있는가?
- [ ] 학습 데이터의 도메인 분포를 확인했는가?
- [ ] 데이터가 최소 요구량을 충족하는가?
- [ ] pretrained 백본을 사용하고 있는가?
- [ ] 아키텍처가 태스크에 적합한가? (히트맵 vs 직접 회귀)
- [ ] 도메인이 다른 데이터를 섞을 때 오버샘플링을 적용했는가?
- [ ] augmentation이 타겟 도메인의 특성을 반영하는가?
- [ ] 학습 데이터 검수가 완료되었는가? (검수 전 학습 금지)
